Go的channel了解吗，Channel的作用是什么，可以存储什么类型的数据

作用是协程之间的数据传输和同步，可以存储任何类型包括内置类型（整数，字符串），自定义类型（接口等）

Channel的底层实现

两个等待队列和锁来实现的

  1. 等待队列： 每个 channel 都有两个相关联的等待队列，一个用于存储发送操作的 goroutine，另一个用于存储接收操作的 goroutine。当一个 goroutine 尝试发送或接收数据时，如果 channel 当前为空（对于接收操作）或已满（对于发送操作），则该 goroutine 将会被加入到相应的等待队列中，直到条件满足。这样可以保证发送和接收操作的顺序性和安全性。
  2. 锁： 除了等待队列，每个 channel 还使用了互斥锁来保护内部数据结构，以确保并发访问的安全性。在进行发送和接收操作时，需要先获取相应的锁，以防止多个 goroutine 同时修改 channel 的状态。

Go的协程调度模型

一个线程管理多个协程，线程称为M（machine），有个线程队列，还有个全局队列，还有个调度器

以下是 Goroutine 调度模型的关键概念和特点：

  1. Goroutine： Goroutine 是 Go 语言并发编程的基本单位。它是一种轻量级的执行体，由 Go 运行时（runtime）管理。与传统的线程相比，Goroutine 的创建和销毁开销很小，可以高效地创建大量的 Goroutine。
  2. 调度器： Go 语言的运行时包含了一个调度器，负责将 Goroutine 安排到操作系统线程上执行。调度器会根据一定的策略将 Goroutine 随机分配给线程，并在需要时进行抢占式调度。
  3. 操作系统线程： 在 Goroutine 调度模型中，操作系统线程由 Go 运行时管理，被称为 M（Machine）。每个 M 都有一个固定大小的 Goroutine 队列，它负责执行队列中的 Goroutine。当一个 Goroutine 阻塞时，M 可以将其从队列中移除，并执行其他可运行的 Goroutine。
  4. 全局运行队列： 除了每个 M 自己的 Goroutine 队列外，调度器还维护了一个全局运行队列（称为 G 队列），用于存放所有可运行的 Goroutine。当某个 M 的 Goroutine 队列为空时，它可以从全局队列中获取 Goroutine 进行执行。

创建一个goroutine时会给它分配什么数据

  1. 栈空间（Stack Space）： 每个 Goroutine 都有自己的栈空间，用于存储局部变量、函数调用信息和其他相关数据。栈空间的大小默认为 2KB，并且可以根据需要自动增长。每个 Goroutine 的栈空间是独立的，因此它们可以同时执行不同的函数调用。
  2. Goroutine 结构体（Goroutine Struct）： 每个 Goroutine 都有一个与之关联的 Goroutine 结构体，其中包含了一些重要的信息，如 Goroutine 的状态、栈空间的起始地址等。这个结构体由 Go 运行时管理，用于跟踪和控制 Goroutine 的执行。
  3. Goroutine 栈指针（Stack Pointer）： Goroutine 栈指针是一个指向当前 Goroutine 栈顶的指针，用于记录当前 Goroutine 的执行位置。当 Goroutine 需要被暂停或切换到其他 Goroutine时，栈指针的值将被保存，并在再次执行该 Goroutine 时恢复。

数据库索引调优

  1. 回表次数太多使用全盘扫描

回表：当对一个列创建索引之后，索引会包含该列的键值及键值对应行所在的rowid。通过索引中记录的rowid访问表中的数据就叫回表。

回表次数太多会严重影响SQL性能，如果回表次数太多，就不应该走索引扫描，应该直接走全表扫描。

  1. 前导模糊查询不能索引
  2. 遵循最左原则

举例来说，假设有一个复合索引 (A, B, C)，那么最左前缀原则要求：

  1. 查询条件中最好包含A字段，可以只包含A字段，也可以同时包含A、B、C字段。

  2. 如果查询条件包含B字段，那么一定要包含A字段，不可以只包含B字段和C字段。

  3. 如果查询条件包含C字段，那么一定要包含A字段和B字段，不可以只包含C字段。

  4. 负向条件查询不能使用索引，可以优化为in查询
  5. 数据库执行计算不会命中索引

数据库底层实现

b+树

哈希索引：哈希索引使用哈希函数将索引键值映射到一个哈希桶（Hash Bucket）中，然后通过哈希桶的指针直接访问对应的数据。哈希索引适用于等值查询，可以在常数时间内找到数据。

然而，由于哈希函数的不确定性，哈希索引不支持范围查询和排序操作。

31. 下一个排列

整数数组的一个 排列 就是将其所有成员以序列或线性顺序排列。

  • 例如，arr = [1,2,3] ，以下这些都可以视作 arr 的排列：[1,2,3]、[1,3,2]、[3,1,2]、[2,3,1] 。

整数数组的 下一个排列 是指其整数的下一个字典序更大的排列。更正式地，如果数组的所有排列根据其字典顺序从小到大排列在一个容器中，那么数组的 下一个排列 就是在这个有序容器中排在它后面的那个排列。如果不存在下一个更大的排列，那么这个数组必须重排为字典序最小的排列（即，其元素按升序排列）。

  • 例如，arr = [1,2,3] 的下一个排列是 [1,3,2] 。
  • 类似地，arr = [2,3,1] 的下一个排列是 [3,1,2] 。
  • 而 arr = [3,2,1] 的下一个排列是 [1,2,3] ，因为 [3,2,1] 不存在一个字典序更大的排列。

给你一个整数数组 nums ，找出 nums 的下一个排列。

session：

  1. 首先用户登录，server 会为用户生成一个 session，为其分配唯一的 sessionId，这个 sessionId 是与某个用户绑定的，也就是说根据此 sessionid（假设为 abc） 可以查询到它到底是哪个用户，然后将此 sessionid 通过 cookie 传给浏览器
  2. 之后浏览器的每次添加购物车请求中只要在 cookie 里带上 sessionId=abc 这一个键值对即可，server 根据 sessionId 找到它对应的用户后，把传过来的商品 id 保存到 server 中对应用户的购物车即可

session问题：大公司用redis存储，防止负载均衡，session找不到了。也就是说假设登录请求打到了 A 机器，A 机器生成了 session 并在 cookie 里添加 sessionId 返回给了浏览器，那么问题来了：下次添加购物车时如果请求打到了 B 或者 C，由于 session 是在 A 机器生成的，此时的 B,C 是找不到 session 的，那么就会发生无法添加购物车的错误，就得重新登录了，此时请问该怎么办。主要有以下三种方式：

1、session 复制

2、session 粘连

这种方式是让每个客户端请求只打到固定的一台机器上，

3、session 共享

这种方式也是目前各大公司普遍采用的方案，将 session 保存在 redis

Token：

可以看到 token 主要由三部分组成

  1. header：指定了签名算法
  2. payload：可以指定用户 id，过期时间等非敏感数据
  3. Signature: 签名，server 根据 header 知道它该用哪种签名算法，再用密钥根据此签名算法对 head + payload 生成签名，这样一个 token 就生成了。
